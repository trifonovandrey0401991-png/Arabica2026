| M8 | Счётчик непрочитанных только растёт | Периодически сбрасывается из loadMessengerUnreadCount |
| M9 | Краш «Кто прочитал» | Cast безопасный — данные уже DateTime |
| M11 | Race condition РКО | Node однопоточный + file locking |
| M15 | Звонки без проверки | Начало звонка проверяется, relay — низкий риск |
| C15 | Штрафы — только JSON | Dual-write гарантирует наличие JSON |

## РЕКОМЕНДУЕМЫЙ ПОРЯДОК ИСПРАВЛЕНИЯ

### Фаза 1 — «Безопасная» (риск МИНИМАЛЬНЫЙ, ~2 часа)
Добавление requireAdmin на 20+ маршрутов. Не ломает ничего, просто закрывает доступ.

**Файлы:** recipes_api, suppliers_api, training_api, envelope_api, shift_handover_questions_api, recount_points_api, rating_wheel_api, order_timeout_api, execution_chain_api, clients_api (broadcast + free-drinks)

**Что может пойти не так:** Если какой-то легитимный сценарий использует эти маршруты от имени не-админа — он перестанет работать. Нужно уточнить у Андрея роли.

### Фаза 2 — «Точечные исправления» (риск НИЗКИЙ, ~3 часа)
Исправление конкретных багов в 1-2 файлах каждый.

**Что делаем:**
- C13: PendingPrizeException — 2 строки
- C8, C9: UTC→MSK в мессенджере — 2 строки
- C10: Lock на голосовании — 5 строк
- H36: Декабрь месяц 13 — 1 строка
- H40: Пустой phone в push — 1 строка
- M1: await canMarkAttendance — 1 слово
- M16, M19, M20: однострочные исправления

**Что может пойти не так:** Минимально. Каждое исправление изолировано.

### Фаза 3 — «Подмена пользователя» (риск СРЕДНИЙ, ~4 часа)
Замена req.body.phone → req.user.phone в 15+ файлах.

**Что делаем:** Проходим по каждому файлу, заменяем источник phone.

**Что может пойти не так:** Flutter может отправлять phone в body, и сервер на него рассчитывает. Нужно проверить каждый Flutter-сервис, который вызывает этот маршрут. Если убрать phone из body — Flutter-сторона может сломаться.

**Как снизить риск:** Не убирать phone из body, а ИГНОРИРОВАТЬ его — всегда брать из req.user.phone. Добавить лог если body.phone ≠ user.phone (для диагностики).

### Фаза 4 — «Двойная запись» (риск СРЕДНИЙ, ~3 часа)
Добавление второй записи (JSON+БД) в 8 модулях.

**Что делаем:** В каждом модуле добавляем запись в недостающее место.

**Что может пойти не так:** Неправильная структура данных при записи в БД (не те имена полей, не тот формат). Нужно проверять по существующим записям в таблице.

### Фаза 5 — «Сложные исправления» (риск ВЫСОКИЙ, ~5 часов)
Требуют аккуратного подхода и тестирования.

| Баг | Почему сложно |
|-----|---------------|
| C1: Заказы не уходят на сервер | Нужен новый метод в OrderService + обработка ошибок + UI feedback |
| H35: Сессии после перезапуска | Затрагивает ВХОД В СИСТЕМУ для ВСЕХ пользователей |
| H37: Утечка сообщений в чате | Нужна фильтрация по участникам — доп. запрос к БД на каждое сообщение |
| H44: hasUnread всегда false | Нужен SQL-запрос — может замедлить список клиентов |
| M5: moscow_time.js | Используется ВЕЗДЕ — неправильное исправление сломает всю систему времени |
| M13: DateTime.now() в 5 файлах | Нужна общая утилита getMoscowDateTime() для Flutter |

**Что может пойти не так:** Поломка входа в систему (H35), поломка всех временных меток (M5), тормоза при загрузке списков (H44).

**Как снизить риск:** Каждое исправление — отдельная ветка, отдельное тестирование. H35 — только после создания резервной копии. M5 — сначала написать тесты.
