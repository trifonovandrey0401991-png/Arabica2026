## Часть 1. Найденные проблемы и точки роста

### A. СЕРВЕР (loyalty-proxy)

#### A1. `rko_api.js` — синхронная операция блокирует сервер
- **Файл:** `loyalty-proxy/api/rko_api.js`, строка 245
- **Суть:** `fs.renameSync()` при загрузке документа РКО блокирует ВЕСЬ сервер (все запросы от всех пользователей стоят) пока файл перемещается
- **Там же:** Метаданные РКО хранятся в одном JSON-файле, который целиком считывается и перезаписывается при каждой загрузке. При 1000+ документов — это тормозит
- **Реальный эффект:** Если два человека одновременно загружают РКО — возможна потеря данных (нет блокировки)

#### A2. `employee_chat_websocket.js` — утечка памяти в typingStatus
- **Файл:** `loyalty-proxy/api/employee_chat_websocket.js`, строки 28-29
- **Суть:** `typingStatus` (карта "кто сейчас печатает") никогда не очищается. Каждый новый чат добавляет записи, старые не удаляются
- **Реальный эффект:** За месяцы работы — тысячи мёртвых записей, сервер потребляет всё больше памяти

#### A3. `clients_api.js` — N+1 запросов к базе в диалогах
- **Файл:** `loyalty-proxy/api/clients_api.js`, строки 970-1010
- **Суть:** Для отображения списка диалогов с клиентами: 1 общий запрос + 2 запроса НА КАЖДОГО клиента. 50 клиентов = 101 SQL запрос
- **Реальный эффект:** Страница диалогов загружается медленно, при росте клиентской базы станет ещё медленнее

#### A4. `pagination.js` — пагинация работает неэффективно
- **Файл:** `loyalty-proxy/utils/pagination.js`, строки 38-56
- **Суть:** Загружает ВСЕ записи из базы в память, затем отрезает нужную страницу через `slice()`. Правильно — брать только нужные записи через SQL `LIMIT/OFFSET`
- **Реальный эффект:** Чем больше данных (отчётов, заказов), тем больше памяти тратится впустую. При тысячах записей — заметное замедление

#### A5. `file_lock.js` — лишняя нагрузка при каждой блокировке
- **Файл:** `loyalty-proxy/utils/file_lock.js`, строка 150
- **Суть:** При каждом захвате блокировки создаётся `new Error().stack` — стек-трейс. Это нужно только для отладки, а выполняется ВСЕГДА
- **Реальный эффект:** Лишняя нагрузка на процессор. Не критично, но тысячи раз в день — складывается

#### A6. `z_report_api.js` — синхронные файловые операции
- **Файл:** `loyalty-proxy/api/z_report_api.js`, строка 498+
- **Суть:** Используются синхронные `fs.writeFile()`, `fs.readdir()`, `fs.unlink()`, `fs.rmdir()` вместо асинхронных
- **Реальный эффект:** При загрузке шаблонов Z-отчётов блокирует обработку других запросов

### B. ПРИЛОЖЕНИЕ (Flutter)

#### B1. Гигантские файлы — сложно поддерживать
- **Масштаб:** 18 файлов больше 1500 строк, 5 файлов больше 2300 строк
- **Самые большие:**
  - `cigarette_training_page.dart` — 4918 строк
  - `main_menu_page.dart` — 3870 строк
  - `shift_handover_questions_management_page.dart` — 2834 строки
  - `my_schedule_page.dart` — 2601 строка
  - `recount_questions_page.dart` — 2477 строк
- **Реальный эффект:** Любое изменение в этих файлах — долго, рискованно, легко сломать соседнюю функцию

#### B2. 272 вызова `setState` без проверки `mounted`
- **Масштаб:** 2014 вызовов `setState`, из них 272 (13.5%) без `if (mounted)`
- **Ключевые файлы:** `main_menu_page.dart` (7 незащищённых), `bonus_penalty_management_page.dart`, `client_functions_page.dart`
- **Реальный эффект:** Если пользователь быстро уходит с экрана во время загрузки данных — приложение может упасть с ошибкой "setState() called after dispose()"

#### B3. `notification_service.dart` — хранит статический BuildContext
- **Файл:** `lib/core/services/notification_service.dart`, строка 44
- **Суть:** `static BuildContext? _globalContext` — контекст экрана хранится как глобальная переменная. Если экран уничтожается, ссылка остаётся
- **Реальный эффект:** Утечка памяти. При длительной работе приложение может потреблять всё больше памяти

#### B4. `base_http_service.dart` — жёсткий лимит 200 записей
- **Файл:** `lib/core/services/base_http_service.dart`, строка 114
- **Суть:** `_defaultPageLimit = '200'` — все списки загружают максимум 200 элементов за раз. Нет кнопки "показать ещё"
- **Реальный эффект:** Если отчётов больше 200 за месяц — старые не видны. Пользователь думает их нет, а они просто не загрузились

#### B5. Сортировка и фильтрация при каждой перерисовке
- **Файлы:** `main_cash_page.dart`, `my_efficiency_page.dart` и другие
- **Суть:** Списки сортируются и фильтруются внутри `build()` — метода который вызывается при каждой перерисовке экрана (десятки раз в секунду)
- **Реальный эффект:** Для 500+ записей — заметные "подтормаживания" интерфейса при прокрутке

#### B6. Утечки памяти — контроллеры без очистки
- **Масштаб:** ~11 файлов с `ScrollController` без `dispose()`, 1 файл создаёт `TextEditingController` в методе диалога без очистки
- **Ключевой пример:** `bonus_penalty_management_page.dart` — `_showAmountDialog()` создаёт контроллеры которые никогда не удаляются
- **Реальный эффект:** Небольшая утечка памяти при каждом открытии диалога

#### B7. `main_menu_page.dart` — слишком много ответственности
- **Файл:** `lib/app/pages/main_menu_page.dart`
- **Суть:** 78 вызовов setState, 15+ параллельных API вызовов при открытии, 12 счётчиков значков, 3 WebSocket-подписки, авто-логин в Telegram-бот, проверка обновлений, запрос GPS
- **Реальный эффект:** Если один из 15 API вызовов падает — рискует сломаться весь главный экран. Медленная загрузка при открытии приложения

#### B8. `Future.delayed` как костыли
- **Масштаб:** 41 вызов `Future.delayed`, из них ~8 используются как "подождать и попробовать снова"
- **Пример:** `main_menu_page.dart` — `Future.delayed(300ms)` перед синхронизацией отчётов
- **Реальный эффект:** Скрывают гонку состояний. Работает "обычно", но при медленном интернете или высокой нагрузке — может сломаться

#### B9. 25+ молчаливых `catch (_) {}` — пустой экран вместо ошибки
- **Файлы:** `manager_grid_page.dart` (2x), `my_dialogs_page.dart` (4x), `messenger_chat_page.dart` (5x), `video_note_recorder_page.dart` (2x) и другие
- **Суть:** Ошибка происходит, но пользователь ничего не видит — экран просто пустой или зависший без объяснений
- **Реальный эффект:** Пользователь думает приложение сломано, перезапускает его. Поддержка не понимает что случилось (нет логов)

#### B10. Нет диалога «Вы уверены?» при выходе из формы
- **Файлы:** `envelope_form_page.dart` (8 полей, 9 шагов), `shift_edit_dialog.dart`, `recount_management_tabs_page.dart`
- **Суть:** Пользователь заполняет форму 20 минут, случайно нажимает «Назад» — ВСЕ данные потеряны без предупреждения
- **Реальный эффект:** Потеря времени сотрудника. Особенно болезненно для длинных форм (конверты, пересчёты)

#### B11. Push-уведомления могут уронить приложение
- **Файл:** `firebase_service.dart`
- **Суть:** Обработчик push не проверяет наличие поля `type` в данных. Если push приходит с нестандартной структурой — краш. Также нет проверки авторизован ли пользователь — push может попытаться открыть экран для сотрудника у клиента
- **Реальный эффект:** Редко, но при нестандартном push — приложение падает

#### B12. Ошибки сети не показываются пользователю
- **Файл:** `base_http_service.dart` и все сервисы
- **Суть:** При потере интернета API запрос молча возвращает пустой список `return []`. Пользователь видит пустой экран без сообщения «Проверьте интернет»
- **Реальный эффект:** Пользователь не понимает что происходит, думает что данных нет

#### B13. Страницы ИИ-обучения загружают сотни фото без лимита
- **Файлы:** `cigarette_training_page.dart`, `coffee_machine_shop_photos_page.dart`
- **Суть:** GridView загружает 200+ фото в полном разрешении. При прокрутке память растёт неограниченно
- **Реальный эффект:** На слабых телефонах — приложение тормозит и может вылететь с ошибкой нехватки памяти

### C. СЕРВЕР — дополнительные находки (глубокий аудит)

#### C1. Номера телефонов в логах без маскировки
- **Файл:** `loyalty-proxy/api/auth_api.js`
- **Суть:** `console.log('✅ User registered: ${normalizedPhone}')` — полный номер телефона виден в логах. Утилита `maskPhone()` уже есть, но не используется в auth_api
- **Реальный эффект:** Персональные данные (телефоны) открыты в логах сервера

#### C2. `/api/efficiency/reports-batch` без ограничения частоты запросов
- **Файл:** `loyalty-proxy/index.js`
- **Суть:** Тяжёлая операция расчёта эффективности (12 SQL запросов) не защищена rate-limiter. Один пользователь может бомбить этот адрес запросами
- **Реальный эффект:** При намеренной или случайной нагрузке — сервер тормозит для всех

#### C3. Бонусы/штрафы можно случайно применить дважды
- **Файл:** `loyalty-proxy/api/bonus_penalties_api.js`
- **Суть:** Нет проверки на повторный запрос (idempotency key). Если сеть зависла и запрос отправлен повторно — бонус/штраф создаётся дважды
- **Реальный эффект:** Сотрудник получает двойной бонус или двойной штраф

#### C4. Некоторые API используют `new Date()` вместо московского времени
- **Файлы:** `rko_api.js`, `shift_transfers_api.js`, `recurring_tasks_api.js`, `employee_chat_api.js`
- **Суть:** `getDate()`, `getMonth()`, `getDay()` вызываются на `new Date()` — это время сервера, а не Москвы. На границе суток (23:00-02:00) даты и месяцы могут определяться неправильно
- **Реальный эффект:** На рубеже месяцев — отчёт может попасть в неправильный месяц. На рубеже суток — дата может отличаться

#### C5. Формат месяца в графике работы не проверяется
- **Файл:** `loyalty-proxy/api/work_schedule_api.js`
- **Суть:** Нет проверки что месяц — корректная строка `YYYY-MM`. Запрос `month=2026-13` создаст файл `2026-13.json` вместо ошибки
- **Реальный эффект:** Низкий — клиент-приложение отправляет правильные значения, но через API можно создать мусорные файлы

### D. ИНФРАСТРУКТУРА — пропущенные проблемы

#### D1. PostgreSQL НЕ включен в резервное копирование
- **Файл:** `loyalty-proxy/backup-script.sh`
- **Суть:** Скрипт копирует 28 JSON-директорий, но НЕ делает `pg_dump` базы данных. С момента миграции (февраль 2026) PostgreSQL — основной источник данных
- **Реальный эффект:** Если база данных повредится — данные потеряны. JSON-файлы как запасная копия могут быть устаревшими

#### D2. Фотографии не копируются
- **Суть:** Директории `shift-photos/`, `employee-photos/`, `recipe-photos/`, `messenger-media/` (~1-10 ГБ) не входят в резервную копию
- **Реальный эффект:** Потеря всех фото при сбое диска

#### D3. 13 таблиц без индексов — медленные запросы
- **Файл:** `loyalty-proxy/utils/db_schema.sql`
- **Ключевые:** `employees` (нет индекса по phone), `clients` (нет индекса по phone), `shops` (нет индекса по address), `coffee_machine_reports` (нет индекса по status), `job_applications` (нет индекса по status)
- **Реальный эффект:** Каждый запрос к сотрудникам/клиентам/магазинам сканирует всю таблицу. При 1000+ клиентов — заметное замедление

#### D4. PM2 не ограничивает потребление памяти
- **Файл:** `loyalty-proxy/ecosystem.config.js`
- **Суть:** Нет настройки `max_memory_restart`. Если происходит утечка памяти — процесс растёт до тех пор, пока система не убьёт его принудительно
- **Реальный эффект:** Сервер может зависнуть на несколько минут пока система пытается освободить память

#### D5. Обновление SSL-сертификата — не автоматизировано
- **Суть:** Нет скрипта автоматического продления сертификата Let's Encrypt. Сертификат действует 90 дней. Если забыть продлить — приложение полностью перестанет работать
- **Реальный эффект:** Все пользователи получают ошибку «Сертификат недействителен», приложение не работает

#### D6. Нет автоматического переключения на JSON при падении базы
- **Файл:** `loyalty-proxy/utils/db.js`
- **Суть:** Если PostgreSQL упадёт — все API начнут возвращать 500 ошибок. Нет автоматического переключения на USE_DB=false
- **Реальный эффект:** Требуется ручное вмешательство для восстановления работы
