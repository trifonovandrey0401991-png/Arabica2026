## 1. Общее описание и жизненный цикл

### Что такое пересменка
Пересменка — процесс передачи смены от одного сотрудника другому. Сотрудник отвечает на вопросы (текст, числа, фото), а управляющая проверяет и ставит оценку (1-10). На основе оценки начисляются баллы эффективности.

### Жизненный цикл отчёта

```
┌──────────┐     ┌──────────┐     ┌──────────────┐
│ PENDING  │────→│  REVIEW  │────→│  CONFIRMED   │
│ Ожидает  │     │На проверке│    │ Подтверждён  │
└──────────┘     └──────────┘     └──────────────┘
     │                │
     │                │           ┌──────────────┐
     │                └──────────→│  REJECTED    │
     │                            │  Отклонён    │
     │                            └──────────────┘
     │
     │           ┌──────────┐     ┌──────────────┐
     └──────────→│  FAILED  │    │   EXPIRED    │
                 │Не пройден│    │  Истёк (>7д) │
                 └──────────┘     └──────────────┘
```

| Статус | Код | Кто меняет | Когда |
|--------|-----|------------|-------|
| **Ожидает** | `pending` | Scheduler (автоматически) | Создаётся по расписанию (утро/вечер) |
| **На проверке** | `review` | Сотрудник (отправляет отчёт) | Сотрудник заполнил вопросы и отправил |
| **Подтверждён** | `confirmed` | Управляющая (ставит оценку) | Управляющая проверила и поставила рейтинг 1-10 |
| **Не пройден** | `failed` | Scheduler (автоматически) | Сотрудник не отправил отчёт до дедлайна |
| **Отклонён** | `rejected` | Scheduler (автоматически) | Управляющая не проверила отчёт в течение `adminReviewTimeout` часов |
| **Истёк** | `expired` | Системная проверка | Отчёт старше 7 дней |

### Типы смен

| Тип | Код | Окно по умолчанию |
|-----|-----|-------------------|
| Утренняя | `morning` | 07:00 — 13:00 |
| Вечерняя | `evening` | 14:00 — 23:00 |

### Модель отчёта (ShiftReport)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | String | Уникальный ID (формат: `{name}_{shop}_{date}_{time}`) |
| `employeeName` | String | Имя сотрудника |
| `employeeId` | String? | ID сотрудника (для связи с графиком) |
| `shopAddress` | String | Адрес магазина |
| `shopName` | String? | Название магазина |
| `shiftType` | String? | Тип смены: `morning` / `evening` |
| `status` | String? | Статус: pending/review/confirmed/failed/rejected/expired |
| `rating` | int? | Оценка от 1 до 10 |
| `answers` | List\<ShiftAnswer\> | Ответы на вопросы |
| `shortages` | List\<ShiftShortage\>? | Список недостач (из DBF остатков) |
| `aiVerificationPassed` | bool? | Результат AI проверки фото |
| `createdAt` | DateTime | Время создания |
| `submittedAt` | DateTime? | Время отправки сотрудником |
| `confirmedAt` | DateTime? | Время подтверждения управляющей |
| `reviewDeadline` | DateTime? | Дедлайн для проверки управляющей |
| `failedAt` | DateTime? | Время провала |
| `rejectedAt` | DateTime? | Время автоматического отклонения |
| `confirmedByAdmin` | String? | Имя управляющей |

## 2. Модуль: Вопросы сдачи смены

### Описание
Вопросы определяют, что именно проверяет сотрудник при пересменке. Вопросы настраиваются управляющей/developer и могут быть привязаны к конкретным магазинам.

### Модель вопроса (ShiftQuestion)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | String | Уникальный ID вопроса |
| `question` | String | Текст вопроса |
| `answerFormatB` | String? | Формат ответа B (free/photo) |
| `answerFormatC` | String? | Формат ответа C (число/number) |
| `shops` | List\<String\>? | Магазины (null = для всех) |
| `referencePhotos` | Map\<String, String\>? | Эталонные фото по магазинам |
| `isAiCheck` | bool | Проверять ли фото через AI |
| `order` | int | Порядок отображения |

### Типы ответов

| Тип | Условие | Что видит сотрудник |
|-----|---------|---------------------|
| **Да/Нет** | `answerFormatB == null && answerFormatC == null` | Кнопки Да/Нет |
| **Число** | `answerFormatC == 'число'` | Числовое поле ввода |
| **Фото** | `answerFormatB == 'free' или 'photo'` | Кнопка камеры |
| **Текст** | Всё остальное | Текстовое поле |

### Связь с пересменкой
```
ShiftQuestion ──загружается──→ ShiftQuestionsPage ──ответы──→ ShiftReport.answers
       │
       ├── shops[] ──фильтрация──→ показываются только вопросы для выбранного магазина
       └── referencePhotos{} ──показывается──→ эталонное фото рядом с вопросом
```

### API

| Метод | Endpoint | Описание |
|-------|----------|----------|
| GET | `/api/shift-questions` | Получить все вопросы |
| GET | `/api/shift-questions?shopAddress=...` | Вопросы для магазина |
| POST | `/api/shift-questions` | Создать вопрос |
| PUT | `/api/shift-questions/:id` | Обновить вопрос |
| DELETE | `/api/shift-questions/:id` | Удалить вопрос |
| PUT | `/api/shift-questions/reorder` | Изменить порядок |

### Хранение на сервере
- Файл: `/var/www/shift-questions/questions.json`

## 3. Модуль: Магазины

### Связь с пересменкой
Каждый отчёт привязан к конкретному магазину через `shopAddress`. Магазин определяет:
- Какие вопросы показывать сотруднику
- Какие эталонные фото использовать
- Кто из управляющих видит отчёт (мультитенантность)

### Взаимодействие

```
ShopService.getShops()
    │
    ├──→ ShiftShopSelectionPage (выбор магазина перед пересменкой)
    ├──→ ShiftReportsListPage (фильтрация отчётов по магазину)
    ├──→ MultitenancyFilterService (фильтрация по managedShopIds)
    └──→ KPI (агрегация по магазину)
```

### Ключевые данные из ShopModel

| Поле | Использование в пересменке |
|------|----------------------------|
| `address` | Главный ключ привязки отчёта к магазину |
| `name` | Отображение в списках и карточках |
| `id` | Связь с managedShopIds в shop-managers.json |

### API
- GET `/api/shops` — список всех магазинов
- Данные: `/var/www/shops.json`

## 4. Модуль: Сотрудники

### Роли и их возможности в пересменке

| Роль | Код | Может заполнять | Может проверять | Видит отчёты |
|------|-----|-----------------|-----------------|--------------|
| **Developer** | `developer` | Нет | Да | Все магазины |
| **Управляющая** | `admin` | Нет | Да | Только свои магазины |
| **Заведующая** | `manager` | Да (свой магазин) | Да (свой магазин) | Свой магазин (или все управляющей) |
| **Сотрудник** | `employee` | Да | Нет | Свой магазин |
| **Клиент** | `client` | Нет | Нет | Нет |

### Связь с пересменкой

```
UserRoleService.loadUserRole()
    │
    ├──→ ShiftQuestionsPage: определяет, может ли заполнять
    ├──→ ShiftReportsListPage: определяет, может ли проверять
    ├──→ ShiftReportViewPage: показывает кнопки подтверждения
    ├──→ MyTasksPage: показывает отчёты на проверке (только admin+)
    └──→ MultitenancyFilterService: фильтрация по магазинам
```

### Модель UserRoleData

| Поле | Тип | Использование |
|------|-----|---------------|
| `role` | UserRole | Определяет доступ |
| `managedShopIds` | List\<String\> | Для admin: какие магазины видит |
| `managedEmployees` | List\<String\> | Для admin: каких сотрудников видит |
| `primaryShopId` | String? | Для manager/employee: основной магазин |
| `canSeeAllManagerShops` | bool | Для manager: видит все магазины управляющей |

### Верификация
Сотрудник должен быть **верифицирован** (через `EmployeeRegistrationService`), иначе он видит приложение как клиент и не может проходить пересменку.

## 5. Модуль: KPI

### Описание
KPI агрегирует данные пересменки в дневную/месячную статистику для отображения на дашбордах.

### Как пересменка попадает в KPI

```
ShiftReportService.getReports()
    │
    └──→ KPIService._loadShiftReports()
            │
            └──→ KPIAggregationService._processShifts()
                    │
                    └──→ KPIDayData (дневная статистика сотрудника)
                            │
                            ├── shiftsCompleted: количество пройденных пересменок
                            ├── shiftRating: средняя оценка
                            └── shiftsMissed: пропущенные пересменки
```

### Фильтрация в KPI

| Фильтр | Описание |
|--------|----------|
| По магазину | `shopAddress` должен совпадать |
| По дате | `createdAt` нормализуется к дате без времени |
| По статусу | Учитываются только `confirmed` и `failed` |

### KPI метрики из пересменки

| Метрика | Источник | Описание |
|---------|----------|----------|
| `shiftsCompleted` | `status == confirmed` | Кол-во подтверждённых пересменок за день |
| `shiftRating` | `report.rating` | Средний рейтинг за день |
| `shiftsMissed` | `status == failed` | Кол-во пропущенных пересменок |

### Страницы KPI, использующие данные пересменки

| Страница | Что показывает |
|----------|----------------|
| `kpi_shop_day_detail_dialog.dart` | Детали дня магазина: список пересменок |
| `kpi_employee_day_detail_page.dart` | Детали дня сотрудника: пересменки |
| `kpi_employee_detail_page.dart` | Месячная статистика сотрудника |
| `kpi_shop_calendar_page.dart` | Календарь магазина с пересменками |

## 6. Модуль: Баллы (Эффективность)

### Формула расчёта баллов за пересменку

```
Рейтинг 1-10  →  Линейная интерполяция  →  Баллы (от minPoints до maxPoints)

Если rating < zeroThreshold:
    points = minPoints + (rating - minRating) / (zeroThreshold - minRating) * (0 - minPoints)

Если rating >= zeroThreshold:
    points = 0 + (rating - zeroThreshold) / (maxRating - zeroThreshold) * (maxPoints - 0)
```

### Настройки по умолчанию

| Параметр | Значение | Описание |
|----------|----------|----------|
| `minPoints` | -3 | Минимум баллов (за рейтинг 1) |
| `zeroThreshold` | 7 | Рейтинг, при котором баллы = 0 |
| `maxPoints` | 2 | Максимум баллов (за рейтинг 10) |
| `missedPenalty` | -3.0 | Штраф за пропуск пересменки |
| `adminReviewTimeout` | 2 (часа) | Дедлайн проверки управляющей |
| `morningStartTime` | 07:00 | Начало утренней смены |
| `morningEndTime` | 13:00 | Конец утренней смены |
| `eveningStartTime` | 14:00 | Начало вечерней смены |
| `eveningEndTime` | 23:00 | Конец вечерней смены |

### Примеры расчёта

| Рейтинг | Баллы | Описание |
|---------|-------|----------|
| 1 | -3.0 | Минимальный рейтинг |
| 4 | -1.5 | Ниже порога |
| 7 | 0.0 | Порог нуля |
| 10 | +2.0 | Максимальный рейтинг |
| Пропуск | -3.0 | Штраф за failed |

### Связь с расчётом

```
ShiftReport (rating: 8)
    │
    └──→ EfficiencyCalculationService.calculateShiftPoints(8)
            │
            └──→ ShiftPointsSettings.calculatePoints(8)
                    │
                    └──→ RatingBasedSettings.calculatePointsFromRating(8)
                            │
                            └──→ Результат: +0.67 балла
```

### API настроек
- GET `/api/points-settings/shift` — получить настройки
- PUT `/api/points-settings/shift` — обновить настройки
- Хранение: `/var/www/points-settings/shift_points.json`

## 7. Модуль: Мои задачи

### Описание
Отчёты пересменки со статусом `review` отображаются в разделе "Мои задачи" для управляющей (admin/developer). Это позволяет быстро перейти к проверке без перехода в модуль пересменки.

### Логика

```
MyTasksPage._loadAssignments()
    │
    ├── Обычные задачи (TaskService)
    ├── Циклические задачи (RecurringTaskService)
    └── Отчёты на проверке (только admin+):
            │
            UserRoleService.loadUserRole()
                │ isAdminOrAbove?
                └──→ ShiftReportService.getReportsForCurrentUser()
                        │
                        └──→ filter: status == 'review'
                                │
                                └──→ _shiftReviewReports (фиолетовые карточки)
```

### Карточка отчёта в "Мои задачи"

| Элемент | Описание |
|---------|----------|
| Иконка | `Icons.rate_review` с фиолетовым градиентом |
| Заголовок | `Пересменка: {shopAddress}` |
| Подзаголовок | Имя сотрудника + дата + тип смены |
| Бейдж | "На проверке" (фиолетовый) |
| По нажатию | Открывает `ShiftReportViewPage` |
| После проверки | Отчёт исчезает из "Мои задачи" автоматически |

### Мультитенантность
`getReportsForCurrentUser()` автоматически фильтрует отчёты по магазинам управляющей — управляющая видит только отчёты из своих магазинов.

## 8. Модуль: Управляющие (Мультитенантность)

### Описание
Мультитенантность определяет, какие данные видит каждый пользователь. Главный источник истины — файл `shop-managers.json` на сервере.

### Структура shop-managers.json

```json
{
  "developers": ["79054443224", "79614761484"],
  "managers": [
    {
      "phone": "79001234567",
      "role": "admin",
      "managedShops": ["shop_001", "shop_002"],
      "employees": ["79111111111", "79222222222"]
    },
    {
      "phone": "79003456789",
      "role": "manager",
      "primaryShopId": "shop_001",
      "canSeeAllManagerShops": true
    }
  ]
}
```

### Цепочка фильтрации для пересменки

```
1. Вход: ShiftReportService.getReportsForCurrentUser()
         │
2.       └──→ BaseReportService.getReportsForCurrentUser()
                │
3.              ├── getReports() — загрузка ВСЕХ отчётов с сервера
                │
4.              └── MultitenancyFilterService.filterByShopAddress()
                        │
5.                      └── getAllowedShopAddresses()
                                │
6.                              └── UserRoleService.loadUserRole()
                                        │
                                        ├── developer → null (видит ВСЁ)
                                        ├── admin → managedShopIds → адреса
                                        ├── manager → primaryShopId → адрес
                                        └── employee → primaryShopId → адрес
```

### Правила фильтрации

| Роль | Что видит | Источник данных |
|------|-----------|-----------------|
| Developer | Все магазины | `developers[]` в shop-managers.json |
| Admin (управляющая) | Только свои магазины | `managedShops[]` в shop-managers.json |
| Admin без привязки | Все магазины | `managedShops` пустой |
| Manager | Свой магазин | `primaryShopId` |
| Manager (расширенный) | Все магазины управляющей | `canSeeAllManagerShops: true` |
| Employee | Свой магазин | `primaryShopId` |

### API
- GET `/api/shop-managers/role/:phone` — получить роль пользователя
- Хранение: `/var/www/shop-managers.json`

## 9. Модуль: Моя эффективность

### Описание
Показывает сотруднику его личную статистику по пересменкам и другим модулям в виде баллов за период.

### Данные пересменки в "Моя эффективность"

```
MyEfficiencyPage
    │
    └── EfficiencyDataService.getEfficiencyData(phone, period)
            │
            └── Сервер вычисляет баллы по efficiency_calc.js
                    │
                    ├── shift_reports → баллы за пересменку
                    ├── shift_reports (failed) → штрафы за пропуск
                    └── Итого за период
```

### Что видит сотрудник

| Метрика | Описание |
|---------|----------|
| Баллы за пересменки | Сумма баллов за подтверждённые отчёты |
| Штрафы за пропуски | Сумма штрафов за failed отчёты |
| Средний рейтинг | Средняя оценка за период |
| Тренд | Динамика по дням/неделям |

### Серверный расчёт (efficiency_calc.js)

| Вход | Формула | Выход |
|------|---------|-------|
| `report.rating` (confirmed) | `calculatePointsFromRating(rating, settings)` | Баллы |
| `report.status == 'failed'` | `settings.missedPenalty` | Штраф (-3.0) |
| `report.status == 'rejected'` | Не учитывается | 0 |

## 10. Модуль: AI верификация

### Описание

### Связь с пересменкой

```
ShiftQuestion (isAiCheck: true)
    │
    └──→ Сотрудник делает фото
            │
            └──→ ShiftAiVerificationService.verify(photo, referencePhoto)
                    │
                    ├── Результат: passed/failed
                    └──→ ShiftReport.aiVerificationPassed = true/false
```

### Модель ShiftAiVerificationModel

| Поле | Описание |
|------|----------|
| `questionId` | ID вопроса |
| `actualPhotoUrl` | URL фото сотрудника |
| `referencePhotoUrl` | URL эталонного фото |
| `result` | passed / failed |
| `confidence` | Уверенность AI (0-1) |
| `differences` | Описание отличий |

### API
- POST `/api/shift-ai-verification/verify` — проверить фото
- Хранение: `/var/www/shift-ai-verification/`

### Страницы

| Страница | Описание |
|----------|----------|
| `shift_ai_verification_page.dart` | Результаты AI проверки |
| `shift_training_page.dart` | Обучение AI на фото пересменки |

## 11. Модуль: Автоматизация (Scheduler)

### Описание
`shift_automation_scheduler.js` — серверный процесс, который автоматически создаёт pending отчёты и обрабатывает дедлайны. Работает по расписанию каждые 5 минут.

### Что делает scheduler

```
Каждые 5 минут:
    │
    ├── 1. createPendingReports()
    │       Для каждого магазина с графиком:
    │       ├── Определить тип смены (morning/evening) по текущему времени
    │       ├── Найти сотрудника из графика работы
    │       └── Создать pending отчёт если ещё нет
    │
    ├── 2. checkSubmissionDeadlines()
    │       Для каждого pending отчёта:
    │       ├── Проверить: прошло ли окно смены?
    │       └── Если да → status = 'failed', failedAt = now
    │
    ├── 3. checkReviewDeadlines()
    │       Для каждого review отчёта:
    │       ├── Проверить: прошёл ли reviewDeadline?
    │       └── Если да → status = 'rejected', rejectedAt = now
    │
    └── 4. applyPenalties()
            Для каждого failed отчёта:
            ├── Начислить штраф (missedPenalty: -3.0)
            └── Отправить push-уведомление сотруднику
```

### Временные зоны
- Scheduler работает в **московском времени (UTC+3)**
- Все дедлайны рассчитываются относительно Москвы

### Хранение
- Отчёты: `/var/www/shift-reports/{YYYY-MM-DD}.json`
- Состояние автоматизации: `/var/www/automation-state/shift-automation.json`

### Блокировка файлов
Scheduler использует `file_lock.js` для предотвращения race conditions при параллельной записи:
```
withLock('/var/www/shift-reports/2026-02-12.json', async () => {
    // читаем → модифицируем → записываем
});
```

## 12. Модуль: Пересменка (Shift Handover)

### Описание
Shift Handover — **отдельный** модуль "Сдача смены", похожий на пересменку, но используется для приёма-передачи кассы, документов и материальных ценностей.

### Отличия от пересменки (Shift)

| Параметр | Shift (Пересменка) | Shift Handover (Сдача смены) |
|----------|-------------------|------------------------------|
| Назначение | Проверка состояния магазина | Передача кассы и материальных ценностей |
| API endpoint | `/api/shift-reports` | `/api/shift-handover-reports` |
| Вопросы | `/api/shift-questions` | `/api/shift-handover-questions` |
| Scheduler | `shift_automation_scheduler.js` | `shift_handover_automation_scheduler.js` |
| Настройки баллов | `shift_points` | `shift_handover_points` |
| Хранение | `/var/www/shift-reports/` | `/var/www/shift-handover-reports/` |
| Роль-выбор | Нет | Да (`ShiftHandoverRoleSelectionPage`) |

### Общее

| Аспект | Общее |
|--------|-------|
| Базовый сервис | `BaseReportService` |
| Мультитенантность | `MultitenancyFilterService` |
| Статусы | pending → review → confirmed/failed/rejected |
| Расчёт баллов | `EfficiencyCalculationService.calculateShiftHandoverPoints()` |
| KPI агрегация | `KPIAggregationService._processShiftHandovers()` |

## 13. Модуль: Уведомления (Push)

### Push-уведомления в пересменке

| Событие | Кто получает | Текст |
|---------|-------------|-------|
| Отчёт подтверждён | Сотрудник | "Ваш отчёт подтверждён. Оценка: X/10" |
| Pending создан | Сотрудник | "Пожалуйста, пройдите пересменку" |
| Failed (пропуск) | Сотрудник | "Пересменка не пройдена. Штраф: -3 балла" |
| Отчёт отправлен | Управляющая | "Новый отчёт пересменки от {name}" |

### Сервис
- `EmployeePushService.sendReportStatusPush()` — отправка push через FCM
- Endpoint: POST `/api/report-notifications/send`
