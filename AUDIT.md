# ЭТАП 3: АНАЛИЗ ПРОИЗВОДИТЕЛЬНОСТИ И ПАМЯТИ

## 3.1 КРИТИЧЕСКИЕ ПРОБЛЕМЫ СЕРВЕРА

### Проблема 1: Синхронные файловые операции (179 вызовов)

**Что это значит простыми словами:**
Когда сервер читает файл "синхронно", он останавливается и ждёт пока файл прочитается. Если 100 пользователей одновременно делают запросы, они все будут ждать по очереди. Это как касса в магазине с одним кассиром - все стоят в очереди.

**Где проблема:**

| Файл | Строки | Проблема |
|------|--------|----------|
| `admin_cache.js` | 79-81, 115-122 | `fs.readdirSync` + `fs.readFileSync` в цикле |
| `index.js` | 664-676, 724-741 | Синхронное чтение всех отчётов |

**Пример плохого кода:**
```javascript
// admin_cache.js:79-81 - БЛОКИРУЕТ СЕРВЕР!
const files = fs.readdirSync(EMPLOYEES_DIR).filter(f => f.endsWith('.json'));
for (const file of files) {
  const content = fs.readFileSync(path.join(EMPLOYEES_DIR, file), 'utf8');
  // При 5000 сотрудниках = 5000 синхронных чтений = сервер "замирает" на 100-500мс
}
```

**Как исправить:**
```javascript
// Правильный вариант - асинхронный
const files = await fsp.readdir(EMPLOYEES_DIR);
const jsonFiles = files.filter(f => f.endsWith('.json'));
const contents = await Promise.all(
  jsonFiles.map(f => fsp.readFile(path.join(EMPLOYEES_DIR, f), 'utf8'))
);
// Все файлы читаются ПАРАЛЛЕЛЬНО, сервер не блокируется
```

### Проблема 2: N+1 паттерн (множественные запросы в цикле)

**Что это значит:**
Вместо одного запроса на 100 записей, делается 100 отдельных запросов. Это в 100 раз медленнее.

**Где проблема:**
```javascript
// backups-refactor/api_clients_api.js:118
const dialogs = files.map(f =>
  JSON.parse(fs.readFileSync(path.join(dialogDir, f), 'utf8'))
);
// 100 диалогов = 100 последовательных чтений файлов
```

### Проблема 3: Race Conditions (гонки данных)

**Что это значит:**
Когда два пользователя одновременно обновляют один файл, один из них может потерять свои изменения.

**Пример:**
```
1. Админ 1 читает сотрудника: { points: 100 }
2. Админ 2 читает сотрудника: { points: 100 }
3. Админ 1 пишет: { points: 110 }  (+10)
4. Админ 2 пишет: { points: 105 }  (+5)
Результат: 105 вместо 115! Изменения Админа 1 потеряны.
```

## 3.2 ПРОБЛЕМЫ FLUTTER

### Проблема 1: FutureBuilder без кэширования

**Что это значит:**
Каждый раз когда экран перестраивается, фотографии загружаются заново вместо использования кэша.

**Где проблема:**
```dart
// shift_report_view_page.dart:369-403
FutureBuilder<String>(
  future: Future.value(PhotoUploadService.getPhotoUrl(answer.photoDriveId!)),
  // Каждый rebuild = новый Future = повторная загрузка фото
)
```

### Проблема 2: Отсутствие const конструкторов

**Что это значит:**
Виджеты пересоздаются при каждом обновлении экрана, хотя могли бы использоваться повторно.

**Пример:**
```dart
// shift_reports_list_page.dart:820-826
Row(
  children: [
    Icon(icon, size: 18, color: Colors.grey),  // Не const
    const SizedBox(width: 8),
    Text(hint, style: const TextStyle(fontSize: 13)),
  ],
)
// При 50 элементах в списке = 50 новых Icon объектов каждый rebuild
```

# ЭТАП 6: МАСШТАБИРУЕМОСТЬ И АРХИТЕКТУРА

## 6.1 КРИТИЧЕСКИЕ АРХИТЕКТУРНЫЕ ПРОБЛЕМЫ

### Проблема 1: Монолит index.js (8,579 строк)

**Что это значит:**
Весь серверный код находится в одном файле. Это как держать все вещи в одной комнате без шкафов - найти что-то сложно, а добавить новое - ещё сложнее.

| Метрика | Значение |
|---------|----------|
| Строк кода | 8,579 |
| HTTP endpoints | 165 |
| fs операций | 179 |
| Использований DATA_DIR | 87 |

**Что будет при росте:**
```
Сейчас: 8,579 строк
+50% функционала: ~13,000 строк (сложно работать)
+100% функционала: ~17,000 строк (невозможно работать)
```

**Решение:** Перенести endpoints в уже существующую папку `api/` (там уже 50 модулей готовы).

### Проблема 2: Файловая система вместо базы данных

**Текущая архитектура:**
```
/var/www/
├── employees/           ← N файлов (каждый сотрудник = файл)
├── shops/              ← N файлов (каждый магазин = файл)
├── shift-reports/      ← N файлов (каждый отчёт = файл)
└── ...                 ← 30+ папок с JSON файлами
```

**Проблемы:**

| Операция | Сейчас | При 10x росте |
|----------|--------|---------------|
| Загрузка всех сотрудников | 500 file reads | 5,000 file reads |
| Поиск по ID | O(N) сканирование | 5,000 сканирований |
| Фильтр по месяцу | Читать ВСЕ файлы | 50,000 операций |

**Решение (долгосрочное):**
- Redis для кэширования горячих данных
- PostgreSQL для транзакционных данных (заказы, штрафы)
- Оставить JSON для холодного хранилища

### Проблема 3: Отсутствие очередей задач

**Сейчас:** 6 планировщиков работают в одном процессе:
- startShiftAutomationScheduler()
- startRecountAutomationScheduler()
- startRkoAutomationScheduler()
- startAttendanceAutomationScheduler()
- startEnvelopeAutomationScheduler()
- startShiftHandoverAutomationScheduler()

**Проблемы:**
- Если один планировщик "завис", все остальные тоже встанут
